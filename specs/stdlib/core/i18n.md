# I18n (Localization)

<!-- quick-info: {"kind":"module","name":"aivi.i18n"} -->
This module provides a small, typed localization surface that fits AIVI's existing primitives:

- `Effect E A` for explicit failure/cancellation.
- Sigils for compile-time validation of structured literals.
- Multiple catalog formats: `.properties`, JSON, and YAML.

This is intentionally a minimal v0.1 foundation: key validation, message template validation, bundle loading, lookup, and rendering.

## What this module is for

`aivi.i18n` helps you keep user-facing text out of application logic while still giving you type-safe tools for loading translations and rendering messages.

In practice, it solves three common problems:

- **Finding the right message** for a locale.
- **Checking message keys and templates early** so mistakes are caught before runtime.
- **Rendering placeholders safely** so translated text can include names, counts, dates, and similar values.

## Start here

If this is your first localized screen, the usual path is:

1. write a key with `~k"..."`,
2. write a message template with `~m"..."`,
3. load a bundle,
4. render the translated message with placeholder values.

The later workflow sections walk through that same order in more detail.

## Message keys and message templates

AIVI gives localization two dedicated sigils so common mistakes are caught where the message is written. Think of them as safer literals: `~k` is for “which message?”, and `~m` is for “what does the translated sentence look like?”.

### `~k"..."` for keys

Use `~k` for translation keys such as `app.welcome`.

<<< ../../snippets/from_md/stdlib/core/i18n/k_keys.aivi{aivi}

`~k` is validated at parse time:

- the key must not be empty,
- segments are separated with dots, like `a.b.c`,
- empty segments are rejected,
- each segment must start with `[A-Za-z_]` and then use only `[A-Za-z0-9_-]`.

### `~m"..."` for messages

Use `~m` for translatable message templates.

<<< ../../snippets/from_md/stdlib/core/i18n/m_messages.aivi{aivi}

`~m` is also validated at parse time. A message can contain:

- plain text,
- escaped braces,
- placeholders such as `{name}` or `{count:Int}`.

### Placeholder syntax

| Form | Meaning |
| --- | --- |
| `{name}` | A named placeholder; any supported renderable type is accepted at render time. |
| `{name:Type}` | A named placeholder with an explicit allowed type. |

In this module, placeholders use a closed allowlist of renderable types instead of a general `ToText` constraint. That keeps message rendering simple and predictable.

In practice, that means message templates stay easy to audit: only a small, known set of value kinds can appear inside placeholders.

Supported placeholder types are:

- `Text`
- `Int`
- `Float`
- `Bool`
- `Decimal`
- `DateTime`

### Escaping braces

Rule of thumb:

- writing a **message template** with `~m` → double the braces
- writing a plain **text literal** → escape the braces with backslashes

There are two brace-escaping rules to remember:

| Context | Literal brace | Example |
| --- | --- | --- |
| Inside `~m"..."` | <code v-pre>{{</code> and <code v-pre>}}</code> | <code v-pre>~m"Price: {{free}}"</code> renders `Price: {free}` |
| Inside a normal `"..."` `Text` literal | `\{` and `\}` | `"Hello, \{name\}"` is the text `Hello, {name}` |
| Source code for `\{` | `\\{` | The backslash itself must be escaped in AIVI source. |

If you are writing a message template as raw `Text` instead of `~m`, the text literal needs the `\{` form:

<<< ../../snippets/from_md/stdlib/core/i18n/escaping_braces_in_text_literals_vs_m_sigils.aivi{aivi}

The sequence `\\{` in source becomes `\{` at runtime, which `message` interprets as a literal brace instead of a placeholder delimiter.

## Supported catalog formats

AIVI can load translations from three flat key/value formats.

### Properties (`.properties`)

```text
app.welcome = Hello, {name:Text}!
app.cartItems = You have {count:Int} items.
```

Lines starting with `#` and blank lines are ignored.

### JSON (`.json`)

```json
{
  "app.welcome": "Hello, {name:Text}!",
  "app.cartItems": "You have {count:Int} items."
}
```

### YAML (`.yaml` / `.yml`)

```yaml
app.welcome: "Hello, {name:Text}!"
app.cartItems: "You have {count:Int} items."
```

For JSON and YAML, the structure is flat in this module. Nested objects and nested maps are not part of the supported format here.

## Tooling: `aivi i18n gen`

The `aivi i18n gen` command turns a translation catalog into a generated AIVI module.

```text
aivi i18n gen <catalog-file> --locale <tag> --module <name> --out <file>
```

That generated module contains:

- `KeyId`, a sum type with one constructor per key,
- `keyText : KeyId -> Text`, which converts the constructor back to its key text,
- `bundle`, a compiled `Map` of `~m"..."` messages,
- `t : KeyId -> {} -> Text`, a shorthand for lookup plus render.

Why use code generation? It replaces stringly-typed keys like `~k"app.welcome"` with constructors such as `AppWelcome`, so typos become compile-time errors.

Example input:

```text
app.welcome = Hello, {name:Text}!
app.goodbye = See you later!
```

## Common workflows

Start here if you want a practical tour. The sections below follow the order most apps need: pick a locale, load translations, render messages, then add fallbacks.

### 1. Determine the user's locale

Use `aivi.system.localeTag` to ask the host system for a best-effort locale tag, then parse it and fall back to a known default.

<<< ../../snippets/from_md/stdlib/core/i18n/1_determine_the_user_s_locale_system_best_effort.aivi{aivi}

Notes:

- On Unix-like systems, `localeTag` checks `LC_ALL`, `LC_MESSAGES`, and `LANG` in that order.
- Suffixes such as `.UTF-8` and `@modifier` are stripped before parsing.
- `parseLocale` accepts either `-` or `_` as separators, so both `en-US` and `en_US` work.
